<objective>
Ship the two safety mechanisms that make the engine/vault boundary structural rather than careful: an allowlist-based `update.sh` (engine→vault sync that copies only explicitly-listed safe paths) and a pre-push hook on the engine repo that aborts a push if any tracked file matches a content-glob safety list. Both are TDD: a test proves the allowlist excludes non-listed files, and a test proves the hook aborts on a glob match.

Purpose: This is the catastrophe-prevention layer (pitfall 4, engine/vault boundary collapse). Worst-case allowlist failure is "forgot to publish a feature" (annoying); a denylist's worst-case is "accidentally published a journal entry" (catastrophic) — so allowlist only, never denylist (D-07). The pre-push hook is the last line of defense before private content reaches the public remote.
Output: update.sh, the allowlist manifest, the pre-push hook, the content-glob list, an install-hooks.sh, and two bash tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| engine source → vault | update.sh writes engine content into the user's private vault; an over-broad copy could overwrite or pull in unintended files |
| engine working tree → public remote | git push moves tracked files to the public GitHub remote; any vault content here is a privacy catastrophe |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | update.sh allowlist false-negative | mitigate | Allowlist, never denylist (D-07): update.sh copies ONLY paths in templates/allowlist.txt; test/update.test.sh proves a non-allowlisted file does not propagate. Worst case is a missing feature, never a leak |
| T-03-02 | Information Disclosure | private content reaching public remote | mitigate | hooks/pre-push scans git ls-files against hooks/content-globs.txt and aborts the push, naming the offender; test proves abort on daily/ and desired.md matches |
| T-03-03 | Tampering | pre-push hook bypassed with git push --no-verify | accept | `git push --no-verify` deliberately skips pre-push hooks — this cannot be prevented at the hook layer and is an intentional developer override. ACCEPTED, and mitigated structurally: .gitignore (plan 01) excludes all vault content paths and the vault is a physically separate working tree, so there is no private content in the engine tree to bypass-push in the first place (DEVELOPMENT.md "why this is mistake-proof") |
| T-03-04 | Tampering | update.sh strict-overwrite wipes vault customizations | accept | D-06: engine version always wins on allowlisted paths; users who want to customize skills must fork the engine and point .engine-remote at their fork. Documented trade-off, not a defect |
| T-03-05 | Elevation of Privilege | unquoted paths in copy/scan loops | mitigate | Quote all path variables; iterate manifest lines safely; no eval of file contents or paths |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan — pure bash + git; nothing to verify |
</threat_model>
