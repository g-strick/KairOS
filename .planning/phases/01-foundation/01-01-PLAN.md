<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input → setup.sh | The vault-path prompt is untrusted free text; a malicious or fat-fingered value could point at an unintended directory |
| engine tree → filesystem | mkdir -p and file copies write to a user-chosen location outside the repo |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | setup.sh vault-path prompt | mitigate | Expand and normalize the path; require explicit confirm screen (D-02) listing the exact target before any write; no `eval` of user input |
| T-01-02 | Denial of Service | setup.sh against existing vault | mitigate | D-03 three-option guard (Overwrite warns, update.sh, Cancel) — never silently overwrite an existing populated vault |
| T-01-03 | Elevation of Privilege | mkdir -p / file copy paths | mitigate | Use quoted variables for all paths; never construct paths via unquoted expansion; create only under the confirmed vault root, never absolute system paths derived from raw input |
| T-01-04 | Information Disclosure | engine repo tracking vault content | mitigate | .gitignore excludes all vault content paths (daily/, goals/, north-star/, inbox.md, habits.md) so private data physically cannot be committed; complemented by plan 03 pre-push hook |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this phase — pure bash + git only; nothing to verify |
</threat_model>
